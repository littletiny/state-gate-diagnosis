# Agent 自主循环与自我改进设计

> 本文档讨论系统的元设计理念：如何让 Agent 实现真正的自主循环和自我改进。
> 
> 注意：本文档只讨论设计哲学，不涉及具体代码实现。

---

## 问题本质：什么是 Agent 的自主性？

### 传统的"伪自主"

大多数所谓的"Agent 系统"实际上是：

```
人类设计流程 → 代码固化逻辑 → Agent 按剧本执行
```

这不是自主，只是自动化。真正的自主性需要：

```
Agent 感知状态 → Agent 决策下一步 → Agent 执行并学习 → 改进自身
           ↑                                              ↓
           └────────────── 循环改进 ──────────────────────┘
```

### 自主性的三个层次

| 层次 | 名称 | 特征 | 示例 |
|------|------|------|------|
| L1 | 任务执行 | 按给定指令完成具体任务 | "分析这个 metric" |
| L2 | 目标驱动 | 根据目标自主规划路径 | "探索连接生命周期" |
| L3 | 元认知 | 改进自己的认知工具 | "发现当前方法不够，发明新方法" |

目标是实现 L3：Agent 不仅能完成任务，还能改进完成任务的方式。

## 自循环设计：永不停歇的探索

### 循环的本质

Agent 的自循环不是简单的"重复执行"，而是：

```
当前状态 ──→ 认知缺口 ──→ 决策行动 ──→ 产生新知 ──→ 更新状态
    ↑                                                    ↓
    └──────────────── 认知螺旋上升 ──────────────────────┘
```

每一个循环周期，Agent 都应该：
1. **重新评估**当前知识体系的状态
2. **识别缺口**（不是人类预设的缺口，而是它自己发现的缺口）
3. **决定**下一步探索什么（不是人类告诉它探索什么）
4. **执行**并产生新知识
5. **反思**这次探索的效果

### 状态感知：Agent 如何"看见"自己？

Agent 需要能够感知自己的知识状态，而不是依赖外部维护的状态文件。

**错误的方式**:
- 人类维护 YAML 状态文件
- Agent 读取预设的"待办清单"

**正确的方式**:
- Agent 实时解析知识库目录
- Agent 自己判断"我还不知道什么"
- Agent 自己评估"什么值得探索"

类比人类学习：你不会依赖一个"待学习清单"，而是根据已掌握的知识，自然发现"我还需要学什么"。

### 缺口识别：从"被动接受"到"主动发现"

缺口识别是自主性的核心。Agent 应该能够：

1. **覆盖度缺口**: "问题场景 X 完全没有 metrics 支持"
2. **深度缺口**: "Metric Y 已探索但只是表层，需要深入机制"
3. **关联缺口**: "Metric A 和 B 明显相关，但关联关系未建立"
4. **应用缺口**: "已探索的知识未转化为可执行的诊断方向"

这些缺口不是人类预先定义的，而是 Agent 通过分析自己的知识库发现的。

## 自我改进设计：Agent 如何进化？

### 改进的对象

Agent 可以改进什么？

| 层级 | 可改进对象 | 说明 |
|------|-----------|------|
| 工具层 | 探索工具 | 如何查询状态、如何选择目标 |
| 方法层 | 工作方法 | 如何分析代码、如何生成文档 |
| 认知层 | 知识组织 | 如何分类 metrics、如何建立关联 |

### 改进的机制：元认知循环

自我改进需要元认知能力——对自己的认知过程进行认知。

```
执行任务 ──→ 反思效果 ──→ 识别方法缺陷 ──→ 设计改进方案 ──→ 验证改进
    ↑                                                            ↓
    └────────────────── 方法进化 ────────────────────────────────┘
```

**具体场景示例**:

- **场景 1**: Agent 发现多次探索相似 metrics 时产出质量不稳定
  - 反思: "我的 prompt 模板不够结构化"
  - 改进: 设计更详细的 Skill 模板，包含具体步骤
  - 验证: 下次使用新模板，对比产出质量

- **场景 2**: Agent 发现 Top-Down 和 Bottom-Up 经常冲突
  - 反思: "我的目标选择策略需要更复杂的优先级算法"
  - 改进: 设计新的缺口评估机制
  - 验证: 观察新策略是否减少冲突

### 改进的载体：Skill 即方法

在我们的设计中，Skill 是方法论的载体。

**Skill 的本质**: 不是"Prompt 模板"，而是"可配置的工作方法"。

```
Skill = 工作方法的外部化
```

Agent 通过以下方式改进 Skill：

1. **创建新 Skill**: 发现新的探索场景需要新的方法
2. **修改现有 Skill**: 发现某个方法不够有效
3. **组合 Skills**: 发现复杂任务需要多阶段方法

**关键设计**: Skill 是可读写的文件，Agent 可以像操作数据一样操作方法。

## 人类在循环中的角色

### 不是"监督者"，而是"启动者"

在真正的自主系统中，人类的角色应该是：

| 角色 | 做什么 | 不做什么 |
|------|--------|---------|
| 启动者 | 设定初始方向、提供基础设施 | 不干预每次迭代 |
| 评估者 | 定期评估产出质量 | 不指导具体方法 |
| 约束者 | 设定边界条件（资源、时间） | 不决定探索路径 |

### 退出机制：何时停止？

Agent 需要知道何时停止。可能的停止条件：

1. **目标达成**: 覆盖了所有预定义的问题场景
2. **边际收益递减**: 连续 N 次迭代无新发现
3. **资源耗尽**: 达到时间/成本上限
4. **外部干预**: 人类要求停止或改变方向

**重要**: 停止条件应该是可配置的，Agent 自己检查这些条件。

## 设计原则总结

### 自主性原则

1. **感知自主**: Agent 自己感知状态，不依赖外部状态文件
2. **决策自主**: Agent 自己决定下一步，不依赖预设流程
3. **改进自主**: Agent 自己改进方法，不依赖人类优化提示词

### 极简原则

1. **代码极简**: 代码只提供基础设施，不提供"智能"
2. **智能外放**: 所有认知工作都交给模型
3. **方法外化**: 方法论体现在可配置的 Skill 中

### 进化原则

1. **元认知**: Agent 能够反思自己的方法
2. **可改进**: 方法（Skill）是可读写的
3. **可验证**: 改进的效果可以被评估

## 与传统系统的对比

| 维度 | 传统 Agent 系统 | 本设计 |
|------|----------------|--------|
| 状态管理 | 外部维护 YAML/数据库 | Agent 实时解析目录 |
| 目标选择 | 人类预设或随机 | Agent 自主发现缺口 |
| 探索方法 | 硬编码在代码中 | 外化为可配置的 Skill |
| 改进方式 | 人类修改代码 | Agent 修改 Skill 文件 |
| 知识组织 | 预设结构 | Agent 自主决定结构 |
| 人类角色 | 监督者、指导者 | 启动者、评估者 |

## 未来的可能性

### Skill 的自动生成

Agent 可能学会：
- 发现当前 Skill 无法处理的新场景
- 分析成功探索的模式
- 生成新的 Skill 来捕获这些方法

### 探索策略的进化

Agent 可能发展出：
- 自己的问题分类体系
- 自己的优先级评估机制
- 自己的探索节奏（何时深入、何时广度优先）

### 协作网络

多个 Agent 可能：
- 各自探索不同领域
- 共享发现的方法（Skills）
- 组合彼此的诊断能力

## 结语

Agent 的自主性不是技术问题，而是**设计哲学问题**。

关键在于：你愿意把多少控制权交给 Agent？

- 如果你相信 Agent 的能力，就应该让它自己感知、自己决策、自己改进。
- 如果你不相信，就会写很多代码来"保护"它，结果只是创建了一个复杂的自动化系统。

设计选择是：**信任，但提供可见性**。

- 我们信任 Agent 能够自主探索
- 但我们提供完整的日志和状态，让人类能够理解和评估
- 我们允许人类在任何时候介入或改变方向

这是一种伙伴关系：Agent 负责执行和进化，人类负责监督和启动。
